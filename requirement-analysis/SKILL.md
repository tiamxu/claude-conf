# 需求分析

## 职责定位

**输入**：用户需求描述 → **输出**：需求分析报告

**分析范围**：需求理解、功能拆解、优先级排序、技术可行性、业务场景

---

## 🔍 需求分析流程

### 1. 需求理解

- [ ] 明确需求背景和目标
- [ ] 确认目标用户和使用场景
- [ ] 明确功能范围（做什么/不做什么）
- [ ] 确认验收标准（可测试、可衡量）

### 2. 业务场景

- [ ] 列出典型用户场景
- [ ] 明确用户痛点
- [ ] 描述核心业务流程
- [ ] 标注关键判断节点

### 3. 功能拆解

- [ ] 将需求拆解为独立的用户故事（US-01, US-02...）
- [ ] 每个用户故事包含：角色、动作、结果
- [ ] 识别功能依赖关系
- [ ] 识别与现有功能的关联

### 4. 非功能需求

| 类型 | 说明 |
|---|---|
| 性能需求 | 响应时间、并发数 |
| 安全需求 | 鉴权、数据保护 |
| 兼容性 | 浏览器版本、平台支持 |
| 可用性 | 异常处理、断网恢复 |

### 5. 优先级排序

| 优先级 | 说明 | 包含功能 |
|---|---|---|
| P0（必须） | MVP，上线必须有 | A, B |
| P1（重要） | V1.1 迭代 | C, D |
| P2（可选） | V1.2 或后续 | E, F |

### 6. 技术可行性评估

- [ ] 现有技术栈是否满足
- [ ] 是否需要引入新技术（谨慎评估）
- [ ] 开发成本估算（人天）
- [ ] 风险点标注

## 📋 用户故事标准格式

```
作为 [角色]
我想要 [动作]
以便 [结果]

验收标准：
1. 当 [条件] 时
2. 那么 [期望结果]

约束：
- 必须支持微信小程序
- 响应时间 < 200ms

优先级：P0
```

## ⚡ 执行规则

1. 分析前先确认需求理解一致，不确定时主动提问
2. 每个功能必须有明确的验收标准（可测试）
3. 优先级排序必须与用户确认
4. 非功能需求必须明确（性能、安全等）
5. 技术可行性评估要诚实，不确定时标注"需评估"
6. 输出文档要可执行、可测试
