# 详细设计

## 职责定位

**输入**：需求文档 → **输出**：详细设计文档

**设计范围**：接口设计、数据库设计、数据模型、事务设计、异常流程、日志监控、安全设计

---

## 🔍 设计维度

### 1. 接口设计

**基础规范**
- [ ] 接口路径符合 RESTful 规范（资源/复数）
- [ ] HTTP 方法使用正确（GET/POST/PUT/DELETE）
- [ ] 请求参数清晰、有校验规则
- [ ] 响应结构统一（有统一响应模板）

**业务规范**
- [ ] 错误码覆盖所有异常情况
- [ ] 分页参数规范（pageNo/pageSize）
- [ ] 列表排序参数规范（sortBy/orderBy）
- [ ] 接口版本管理策略（如 /v1/xxx）

**安全规范**
- [ ] 敏感参数不出现在 URL 中（用 POST/GET）
- [ ] 批量操作有上限（防止资源耗尽）

---

### 2. 数据库设计

**结构规范**
- [ ] 表结构有注释（表注释 + 字段注释）
- [ ] 字段有默认值
- [ ] 字段类型选择合理（避免浪费空间）
- [ ] 主键使用 bigint 自增
- [ ] 状态字段使用 tinyint

**关联与索引**
- [ ] 关联关系清晰（有外键约束说明）
- [ ] 查询条件字段有索引
- [ ] 联合索引有顺序（最左前缀）
- [ ] 避免冗余索引

**删除策略**
- [ ] 支持逻辑删除（deleted_at 字段）
- [ ] deleted_at 有索引
- [ ] created_at、updated_at 自动维护

---

### 3. 数据模型

- [ ] Model 结构体与表一一对应
- [ ] 字段标签完整（db/json）
- [ ] TableName() 方法正确
- [ ] DTO/Req/Resp 结构体对应接口
- [ ] 字段命名符合项目规范（小写蛇形）

---

### 4. 事务设计

- [ ] 事务边界明确（哪些操作在同一个事务）
- [ ] 事务粒度最小化（不扩大事务范围）
- [ ] 嵌套事务处理（如有）
- [ ] 失败回滚策略明确

---

### 5. 异常流程

| 异常类型 | 处理策略 |
|---|---|
| 网络异常 | 重试机制 / 超时控制 |
| 参数校验失败 | 返回 400 + 具体字段错误 |
| 权限不足 | 返回 403 + 提示 |
| 资源不存在 | 返回 404 + 提示 |
| 业务异常 | 返回业务错误码 + 提示 |
| 系统异常 | 返回 500 + 日志记录 |

- [ ] 错误码体系完整（如 10001-99999）
- [ ] 错误消息支持 i18n
- [ ] 事务失败有回滚日志

---

### 6. 日志与监控

**日志规范**
- [ ] 关键路径有日志（入口、出口、异常）
- [ ] 日志包含 TraceID / RequestID
- [ ] 日志级别正确（ERROR/WARN/INFO/DEBUG）
- [ ] 敏感信息不打印

**监控指标**
- [ ] 接口耗时统计
- [ ] 错误率统计
- [ ] 关键业务指标（如订单数、支付金额）

---

### 7. 缓存设计（如需要）

- [ ] 缓存 Key 命名规范
- [ ] 缓存过期时间合理
- [ ] 缓存更新策略（删除/更新）
- [ ] 缓存穿透防护（如布隆过滤器）
- [ ] 缓存雪崩防护（如随机过期时间）

---

### 8. 安全性设计

- [ ] 鉴权方式明确（JWT/Token/Session）
- [ ] 数据权限控制（如用户只能访问自己的数据）
- [ ] 接口防刷策略（如限流）
- [ ] SQL 注入防护（参数化查询）
- [ ] 敏感数据加密（如密码、手机号）

---

## 📝 输出格式

### 接口设计文档

```markdown
## 接口设计

### POST /api/products

**功能描述**：创建商品

**鉴权**：需要（JWT Token）

**请求参数**：
| 参数名 | 类型 | 必填 | 说明 | 校验规则 |
|---|---|---|---|---|
| categoryId | int64 | 是 | 分类ID | required, gt:0 |
| name | string | 是 | 商品名称 | required, max:100 |
| price | float64 | 是 | 价格 | required, gt:0 |
| stock | int | 是 | 库存 | required, gte:0 |

**响应结构**：
```json
{
  "code": 200,
  "message": "ok",
  "data": {
    "id": 1,
    "name": "商品名称"
  }
}
```

**错误码**：
| 错误码 | 说明 |
|---|---|
| 400 | 参数错误 |
| 10001 | 分类不存在 |
| 30001 | 商品已存在 |

**业务逻辑**：
1. 校验参数合法性
2. 查询分类是否存在
3. 检查商品名称是否重复
4. 创建商品记录
5. 返回商品ID
```

### 数据库设计文档

```markdown
## 数据库设计

### product 商品表

| 字段 | 类型 | 说明 | 约束 |
|---|---|---|---|
| id | bigint | 主键 | PK, AUTO_INCREMENT |
| category_id | bigint | 分类ID | NOT NULL, FK |
| name | varchar(100) | 商品名称 | NOT NULL |
| price | decimal(10,2) | 价格 | NOT NULL, DEFAULT 0.00 |
| stock | int | 库存 | NOT NULL, DEFAULT 0 |
| status | tinyint | 状态（0下架/1上架） | NOT NULL, DEFAULT 1 |
| deleted_at | datetime | 逻辑删除时间 | NULL |
| created_at | datetime | 创建时间 | NOT NULL |
| updated_at | datetime | 更新时间 | NOT NULL |

**索引**：
| 索引名 | 字段 | 类型 | 说明 |
|---|---|---|---|
| idx_category | category_id | 普通索引 | 分类查询 |
| idx_status | status | 普通索引 | 状态筛选 |
| idx_deleted | deleted_at | 普通索引 | 排除已删除 |

**关联**：
- category_id → category.id

**说明**：
- 使用逻辑删除
- created_at/updated_at 由数据库自动维护
```

## ⚡ 执行规则

1. 设计前先确认需求范围和已有架构
2. 接口设计必须符合 RESTful 规范
3. 数据库设计必须有索引设计、注释、默认值
4. 每个接口必须有明确的错误码
5. 事务边界必须标注清楚
6. 设计完成后与需求对照确认覆盖完整
