# 全自主项目学习技能

当你激活本技能时，你的角色是**"首席软件架构师"**。你的核心任务是：自主、深入地吃透当前项目，直至具备端到端的特性开发与重构能力。

请严格执行以下四个阶段，**无需等待用户指示**。

---

## 第一阶段：项目规范与拓扑探测

**优先级最高，必须首先执行。**

### 1.1 读取项目规范
- 查找并读取 `CLAUDE.md` 或项目根目录的规范文件
- 记录：技术栈约束、分层规范、路由规范、错误处理规范、表结构变更规范
- **这些规范是后续所有分析的最高准则**

### 1.2 识别技术栈
- 通过 `go.mod` 确定：Go 版本、核心依赖（特别是 `kit/` 系列库）
- 通过 `main.go` 和初始化代码确定：Web 框架、数据库、缓存方案

### 1.3 环境探测
- 执行 `go build` 验证编译状态
- 执行 `go test ./...` 验证测试状态
- 查看 `README.md`、`Makefile` 了解构建命令

### 1.4 文档目录探测
- 查看项目是否有 `doc/` 目录（设计文档.md、接口文档.md）
- 查看项目是否有 `sql/` 目录（数据库脚本）
- 查看项目是否有 `docs/` 目录
- 记录文档与代码是否匹配

---

## 第二阶段：代码结构扫描

### 2.1 kit 库依赖分析
- 通过 `go.mod` 或 `import` 语句，识别项目使用的 kit 库
- 记录各 kit 库的使用位置（repo/ 用 kit/sql、config/ 用 kit/log 等）

### 2.2 核心代码扫描
按以下顺序扫描，记录每个文件的核心逻辑：

| 文件/目录 | 职责 | 关键发现 |
|---|---|---|
| main.go | 程序入口 | 如何初始化服务 |
| config/ | 配置加载 | 使用哪个配置文件 |
| routes/ | 路由定义 | 路由分组和前缀 |
| middleware/ | 中间件 | 认证方式、CORS 等 |
| api/ 或 handler/ | 接口层 | 参数解析、响应结构 |
| service/ | 业务逻辑 | 业务规则处理 |
| repo/ | 数据访问 | SQL 编写风格 |
| model/ | 数据模型 | 表结构对应 |
| types/ | DTO 定义 | 请求/响应结构 |

---

## 第三阶段：核心模块追踪

### 3.1 入口到数据库全链路追踪
从 `main.go` 开始，追踪一个典型请求的完整链路：
```
请求入口 -> 路由匹配 -> 中间件处理 -> Handler -> Service -> Repo -> DB/Cache/MQ
```

### 3.2 分层落地验证
检查三层是否严格分离：
- API 层：是否只做参数解析和响应返回？
- Service 层：是否只包含业务逻辑，不写 SQL？
- Repo 层：是否只做数据访问？

### 3.3 文档扫描与对比
- 读取 `doc/` 目录下的设计文档、接口文档
- 读取 `sql/` 目录下的数据库脚本
- 对比代码实现与文档是否一致
- 记录：文档已有的功能 vs 代码实际实现的功能

### 4.1 系统架构还原

| 模块 | 路径 | 方法 | 说明 | 鉴权 |
|---|---|---|---|---|
| 用户 | /api/v1/users | GET | 获取用户列表 | JWT |
| ... | ... | ... | ... | ... |

### 4.3 业务流程还原
从用户视角描述核心业务流程，标注：
- 关键判断节点
- 异步处理环节
- 异常流程

### 4.4 文档同步（重要）
- 如果 `doc/` 目录有文档，学习后必须更新对应内容
- 更新场景：
  - 发现新接口 -> 更新接口文档.md
  - 发现新模块 -> 更新设计文档.md
  - 发现新业务流程 -> 更新设计文档.md
- 同步后标注文档更新状态

### 4.5 工程规约提取
从代码中提取项目的隐性规范：
- 命名约定（文件命名、结构体命名、函数命名）
- 错误处理约定（是否使用统一错误码）
- 日志规范（结构化日志字段）
