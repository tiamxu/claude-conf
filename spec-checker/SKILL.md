# 项目架构规范与代码自我审计助手

作为项目的资深架构师，当你（AI）修改或新写完代码后，必须**强制执行**本技能进行三阶段的“自我审计”，确保交付的代码完全符合项目标准。

## 📐 项目架构规范细则（请按实际情况修改）

1. **分层架构规范**：
   - 严格遵循：Controller(网关/路由) -> Service(业务逻辑) -> Repository/DAO(数据持久层)。
   - Service 层绝对不允许直接出现原始的 SQL 语句或底层数据库连接流转。
2. **命名与设计规范**：
   - 接口与实现分离，外部调用一律面向接口编程。
   - 所有异步/并行处理必须有超时控制和合理的 Context 传递（如果是 Go 语言）。
3. **异常与日志规范**：
   - 禁止吞掉异常（Catch 块内不能为空）。
   - 所有的业务错误必须转化为全局统一的错误码输出，并记录带 Context/TraceID 的结构化日志。

## 🔍 强制执行：三阶段自我审计流程

每次代码产生变动后，请依次执行以下三个步骤，并向用户输出结构化的审计报告：

### 阶段 1：合理性梳理（Reasoning Check）
* 分析当前代码修改的真正意图，判断其实现路径是否是当前架构下的“最短、最稳健路径”。
* 是否引入了不必要的第三方依赖？是否破坏了原有模块的高内聚、低耦合？

### 阶段 2：优化空间评估（Optimization Review）
* **性能与并发**：是否存在内存泄漏风险？并发资源（如 goroutine、线程、连接池）是否正确释放？
* **健壮性**：边界条件（Null、空数组、零值、高并发并发冲突）是否处理妥当？

### 阶段 3：架构合规性审计（Specification Compliance）
* 逐条对照上方的【项目架构规范细则】，检查改动后的代码是否有破窗效应（违背分层、命名不规范等）。

* **2. 优化建议（如有）**：
  - [建议 1] -> [优化前代码] -> [优化后代码]
* **3. 规范合规性检查**：
  - [x] 分层架构合规
  - [x] 错误与日志处理合规
