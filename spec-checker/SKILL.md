# 代码审查

## 两种模式

| 模式 | 命令 | 说明 |
|---|---|---|
| 增量检查 | `/spec-checker` | 检查用户提供的文件列表 |
| 全量扫描 | `/spec-checker full` | 对整个项目进行全面扫描 |

---

## 使用方法

### 增量检查
```
/spec-checker
或
/spec-checker file1.go file2.go
```

用户说明要检查的文件，AI 直接读取这些文件进行审查。

### 全量扫描
```
/spec-checker full
或
/spec-checker full /path/to/project
```

AI 直接扫描项目目录下所有代码文件。

## 🔍 固定检查点（6大维度）

### 1. 命名规范

**Go**
- [ ] 文件：`小写蛇形.go`（如 `product.go`、`admin_auth.go`）
- [ ] 结构体：大写驼峰（如 `Product`、`ProductI18n`）
- [ ] 错误码：`Err` 前缀（如 `ErrProductNotFound`）+ `GetMsg()` 函数
- [ ] 请求/响应：`types/req.go`、`types/resp.go`

**uni-app / 小程序**
- [ ] 组件：PascalCase（如 `ProductCard.vue`）
- [ ] 页面目录：kebab-case（如 `pages/product-detail/`）
- [ ] API 文件：小写蛇形（如 `product.ts`、`user.ts`）

**Vue / Web**
- [ ] 组件：PascalCase（如 `ProductCard.vue`）
- [ ] 视图目录：kebab-case（如 `views/product/` 内含 `list.vue`）
- [ ] Store 文件：小写蛇形（如 `user.js`、`product.js`）
- [ ] API 文件：小写蛇形（如 `user.js`、`product.js`）

---

### 2. 分层规范

**Go**
- [ ] `api/`：仅做参数解析、调用 Service、返回响应
- [ ] `service/`：仅含业务逻辑，不写 SQL
- [ ] `repo/`：仅做数据访问，不含业务逻辑
- [ ] `model/`：数据库结构体（与表一一对应）
- [ ] `types/`：请求/响应结构体

**uni-app / 小程序**
- [ ] `/api/`：统一封装接口（request.ts 基础实例，各模块 *.ts）
- [ ] `/pages/`：仅做 UI 渲染，不含复杂业务逻辑
- [ ] `/store/`：状态管理
- [ ] `/components/`：纯展示组件

**Vue / Web**
- [ ] `/api/`：统一封装接口（base.js 基础实例，各模块 *.js）
- [ ] `/views/`：仅做 UI 渲染和交互
- [ ] `/stores/`：仅做状态管理
- [ ] `/components/`：纯展示组件，业务逻辑抽到 `/composables/` 或 `/utils/`

---

### 3. 安全性

- [ ] 无硬编码密钥/密码/Token
- [ ] 配置用环境变量 `${DB_PASSWORD}`
- [ ] SQL 使用参数化查询，无字符串拼接
- [ ] 需鉴权接口有中间件挂载
- [ ] 日志中无敏感信息
- [ ] 错误统一处理，无裸 error 返回给客户端

---

### 4. 模块化

- [ ] 函数长度 ≤ 80 行
- [ ] 无重复代码（超过 3 处应抽象）
- [ ] 组件可复用、props 清晰
- [ ] 工具函数抽到 utils/
- [ ] 边界值有处理（nil/空/零值）
- [ ] 日志记录关键路径（入口、出口、异常）

---

### 5. 性能安全

**Go**
- [ ] 循环内无数据库查询
- [ ] goroutine 有超时控制
- [ ] 连接池配置合理

**uni-app / 小程序**
- [ ] 避免频繁 setData
- [ ] 大列表使用分页
- [ ] storage 容量限制（10MB）

**Vue / Web**
- [ ] 避免在模板中使用复杂计算（用 computed）
- [ ] 大列表使用虚拟滚动
- [ ] 合理使用 watch，避免滥用
- [ ] 图片懒加载
- [ ] 组件按需加载

---

### 6. 架构与设计

**功能设计**
- [ ] 接口设计是否符合单一职责（一个接口做一件事）
- [ ] 数据模型是否合理（字段冗余、关联关系）
- [ ] 是否存在重复功能（相同逻辑多处实现）

**代码设计**
- [ ] 分层是否清晰（是否跨层调用、循环依赖）
- [ ] 扩展性是否良好（新增功能是否需要改很多地方）
- [ ] 事务边界是否正确（事务范围是否合理）

**可优化项**
- [ ] 重复逻辑是否可抽象为公共函数
- [ ] 过长函数是否可拆分
- [ ] 硬编码是否可抽为配置或常量

---

### 6. 接口级检查（具体到每个接口）

**Go API 层**
- [ ] 接口路径是否符合 RESTful 规范
- [ ] 参数解析是否正确（binding 标签）
- [ ] 错误处理是否统一（使用 pkg/e）
- [ ] 响应结构是否统一

**Go Service 层**
- [ ] 每个函数的逻辑是否清晰
- [ ] 错误是否正确包装
- [ ] 是否有空指针风险（nil 检查）
- [ ] 事务边界是否正确

**Go Repo 层**
- [ ] SQL 是否使用参数化查询
- [ ] WHERE 条件字段是否有索引
- [ ] 是否存在 SQL 注入风险
- [ ] 分页查询是否合理

**Vue / uni-app 组件**
- [ ] 接口调用是否在 api/ 目录
- [ ] 组件是否有适当的 props 校验
- [ ] 是否有未清理的副作用（定时器、事件监听）
- [ ] setData 数据量是否过大

**逐文件检查**
- [ ] 每个文件是否有明确的职责
- [ ] 是否有行数超过 100 行的函数
- [ ] 是否有重复代码模式

## ⚡ 执行规则

1. 增量检查：读取用户提供的文件，不使用 git 命令
2. 全量扫描：直接扫描项目目录下所有代码文件，不使用 git 命令
3. 检查点固定为 6 大维度 + 接口级检查，不得增减
4. 每项必须有明确结论（✅ 通过 / ❌ 不通过）
5. 问题清单需包含文件位置和行号
6. 每个问题必须给出具体的优化方案
