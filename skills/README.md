# Skill 使用流程指南

## 🎯 完整开发流程

```
需求阶段 → 设计阶段 → 架构阶段 → 开发阶段 → 上线阶段
    ↓           ↓           ↓           ↓           ↓
require    design      architect   spec        部署上线
-ment      -document   -review    -checker
-analysis
```

---

## 📋 Skill 一览

| Skill | 命令 | 触发时机 | 模式 |
|---|---|---|---|
| 需求分析 | `/requirement-analysis` | 需求提出时 | 交互式 |
| 详细设计 | `/design-document` | 需求确认后 | 文档生成 |
| 架构评审 | `/architect-review` | 设计文档完成后 | 评审式 |
| 代码审查 | `/spec-checker` | 每次代码修改后 | 增量检查 |
| 全量审查 | `/spec-checker full` | 开发完成后 | 全量扫描 |
| 项目学习 | `/project-learner` | 新项目首次接触 | 快速了解 / 深度审计 |
| 项目初始化 | `/go-project-init` | 初始化新 Go 项目 | 脚手架生成 |

---

## 🔄 各阶段详细说明

### 阶段一：需求设计

**触发**：`/requirement-analysis`

**输入**：用户描述的原始需求

**输出**：需求分析报告

```
用户：我要实现一个商品收藏功能

↓

AI 询问：
- 需求背景和目标是什么？
- 目标用户是谁？
- 功能范围有哪些？
- 验收标准是什么？

↓ 用户回答后

AI 产出：
1. 需求背景
2. 用户故事（US-01, US-02...）
3. 功能拆解
4. 优先级排序（ MVP / V1.1 / V1.2）
5. 技术可行性评估
```

---

### 阶段二：详细设计

**触发**：`/design-document`

**输入**：需求文档

**输出**：详细设计文档（接口文档 + 数据库文档）

```
AI 产出：
1. 接口设计（路径、方法、参数、响应、错误码）
2. 数据库设计（表结构、索引、关联）
3. 数据模型（Model、DTO）
4. 事务设计
5. 异常流程处理策略
6. 日志与监控设计
7. 缓存设计（如需要）
8. 安全性设计
```

---

### 阶段三：架构评审

**触发**：`/architect-review`

**输入**：详细设计文档

**输出**：架构评审报告

```
AI 从 7 个维度评审：
1. 业务理解
2. 架构设计
3. 技术实现
4. 安全性
5. 性能可靠性
6. 可维护性
7. 成本评估

AI 输出：
1. 各维度评估结论
2. 风险清单（🔴高/🟡中/🟢低）
3. 备选方案（如涉及重要决策）
4. 最终结论：✅可行 / ⚠️需优化 / ❌不可行
```

---

### 阶段四：开发编码

**触发**：AI 直接开发 + `/spec-checker`

**输入**：设计文档

**输出**：代码 + 审查报告

```
1. AI 编写代码
2. AI 或用户触发 /spec-checker
3. AI 产出审查报告

审查维度：
1. 命名规范
2. 分层规范
3. 安全性
4. 模块化
5. 性能安全
6. 架构与设计
```

---

### 阶段五：全量审查

**触发**：`/spec-checker full`

**输入**：全部代码

**输出**：全量审查报告

```
1. 执行 /spec-checker full
2. AI 对全项目进行扫描
3. AI 按模块输出检查结果
4. 用户修复所有 ❌ 问题
5. 最终确认通过
```

## 📌 项目学习

**触发**：`/project-learner`

**输入**：无

**输出**：项目认知报告

```
使用场景：
- 新项目首次接触
- 需要了解项目架构
- 需要全局代码审计

输出内容：
1. 技术栈与拓扑
2. 系统架构图
3. 接口清单
4. 核心业务流程
5. 项目规约与风险
```

## ⚡ 标准开发流程示例

```
用户：我要实现商品收藏功能

↓ /requirement-analysis
AI：
1. 询问需求细节
2. 用户回答
3. 产出需求分析报告

↓ 用户确认需求文档

↓ /design-document
AI：
1. 设计接口文档
2. 设计数据库文档
3. 产出详细设计文档

↓ 用户确认设计文档

↓ /architect-review
AI：
1. 7维度评审
2. 产出评审报告
3. 备选方案（如需要）

↓ 评审通过

↓ 开始开发
AI：编写代码
↓ /spec-checker（增量检查）
AI：产出审查报告
用户：修复问题
↓ 重复直到通过

↓ /spec-checker full（全量扫描）
AI：最终审查报告

↓ 通过
↓ 上线
```

## 🎯 流程选择

| 场景 | 推荐流程 |
|---|---|
| 新功能开发 | ✅ 完整流程（需求→设计→架构→开发→审查） |
| 小改动/修复 | 简化：直接开发 → `/spec-checker` |
| 需求不清晰 | ✅ 必须先 `/requirement-analysis` |
| 架构不确定 | ✅ 必须先 `/architect-review` |
| 新项目首次接触 | ✅ 先 `/project-learner` |
