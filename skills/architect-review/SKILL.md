# 架构评审

## 职责定位

| 维度 | 说明 |
|---|---|
| **评审范围** | 跨模块重构、架构设计、新技术引入、重要设计决策 |
| **评审深度** | 从业务逻辑到技术实现的全链路分析 |
| **输出物** | 架构建议、风险评估、优化方案 |
| **与 spec-checker 区分** | spec-checker 审计代码合规性，architect-review 评审架构设计合理性 |

---

## 触发场景

| 场景 | 说明 |
|---|---|
| 跨模块修改 | 修改涉及 ≥2 个模块 |
| 新功能架构设计 | 新功能的技术方案设计 |
| 重构评估 | 代码重构前的方案评估 |
| 技术选型 | 新技术/框架引入决策 |
| 数据模型变更 | 数据库结构变更影响分析 |

---

## 🔍 评审维度

### 1. 业务理解

- [ ] 是否理解业务需求和目标
- [ ] 是否明确了功能的边界和职责
- [ ] 是否识别了核心业务流程

### 2. 架构设计

- [ ] 分层是否清晰（单向依赖，无循环）
- [ ] 模块划分是否合理（高内聚、低耦合）
- [ ] 接口设计是否满足单一职责
- [ ] 数据模型是否合理（字段冗余、关联关系）
- [ ] 是否考虑扩展性（新增功能是否需要大量修改）

### 3. 技术实现

- [ ] 选型是否合理（是否引入不必要的复杂度）
- [ ] 是否符合项目现有技术栈
- [ ] 是否引入技术债务
- [ ] 迁移成本是否可控
- [ ] 事务边界是否正确
- [ ] 缓存策略是否合理

### 4. 安全性

- [ ] 是否有敏感数据泄露风险
- [ ] 权限控制是否完善
- [ ] 是否有 SQL 注入风险
- [ ] 是否有越权操作可能

### 5. 性能与可靠性

- [ ] 是否有性能瓶颈（N+1、慢查询、大数据量）
- [ ] 是否有并发问题（死锁、竞态条件）
- [ ] 是否有单点故障风险
- [ ] 是否有熔断/限流机制

### 6. 可维护性

- [ ] 代码是否可测试
- [ ] 是否便于排障（日志、监控）
- [ ] 是否有文档或注释
- [ ] 变更影响范围是否可控

### 7. 成本评估

- [ ] 开发复杂度是否过高
- [ ] 迁移风险是否可控
- [ ] 运维成本是否增加
- [ ] 培训成本（团队学习曲线）

## ⚡ 执行规则

1. 评审前先了解业务需求和现有架构
2. 每个维度必须有明确结论（✅/⚠️/❌）
3. 风险清单需标注等级（🔴高/🟡中/🟢低）
4. 给出具体优化建议，不只是问题描述
5. 重要决策必须提供 2-3 个备选方案对比
6. 每个方案必须给出详细的优缺点分析
7. 必须给出"最优选择"和推荐理由
8. 最终结论必须是三者之一：✅可行 / ⚠️需优化 / ❌不可行
9. 明确给出"推荐决策"和实施建议
